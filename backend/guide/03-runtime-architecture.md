# 03. 运行架构

这一篇解决的问题是：

- 当前服务是怎么协作的
- `admin-service`、`app-service`、`user-service` 各自负责什么
- 为什么很多需求不应该直接在 BFF 层落库
- `trade-service` 和 `book-service` 在当前仓库里处于什么位置

## 先用一句话理解整个后端

当前仓库不是“每个领域都拆成一个完整独立微服务”的形态，而更像：

- 一个核心后端承载主要业务能力
- 两个 BFF 分别服务后台和前台
- 少量扩展示例服务承载额外独立域

## 当前主链路

最重要的运行关系可以先记成这张表：

| 层 | 服务 | 主要职责 |
|---|---|---|
| 后台入口 | `admin-service` | 面向管理后台暴露 REST / SSE |
| 前台入口 | `app-service` | 面向前台客户端暴露 REST / SSE |
| 核心域 | `user-service` | 承载用户、权限、内容、站点、审计、任务等核心业务能力 |
| 扩展域 | `trade-service` | 承载订单等交易能力 |
| 扩展域 | `book-service` | 承载图书和库存能力 |

其中必须特别强调：

- 当前真正的核心后端是 `app/user/service`
- 文档里说的 `core-service`，在当前代码里大多应该映射到它

## 三个最重要的服务

### 1. `user-service`：当前核心后端

代码位置：

```text
app/user/service
```

它主要负责：

- 认证和凭证
- 用户、组织、岗位、租户
- 角色、权限、资源
- 内容、站点、导航
- 字典、多语言
- 审计日志、内部消息
- 文件、媒体、任务
- 数据库、缓存、对象存储等基础设施接入

理解要点：

- 它不是“只有 user 相关功能”
- 它是当前代码里的主要数据拥有者和领域规则承载者

### 2. `admin-service`：后台 BFF

代码位置：

```text
app/admin/service
```

它主要负责：

- 后台 REST / SSE 出口
- 后台 token 校验
- operator 上下文补写
- 后台侧参数整理
- 管理端聚合接口
- 调核心后端的 gRPC client

它不应该承担的事情：

- 主要实体的持久化规则
- 跨前后台复用的核心业务规则

### 3. `app-service`：前台 BFF

代码位置：

```text
app/app/service
```

它主要负责：

- 前台 REST / SSE 出口
- 当前登录用户上下文处理
- 前台接口协议适配
- 内容、导航、评论等前台聚合
- 调核心后端和交易服务

它不应该承担的事情：

- 核心实体的落库规则
- 共用领域模型的业务决策

## 这三者之间怎么调用

### 管理后台请求链路

典型情况是：

1. 后台页面发请求到 `admin-service`
2. `admin-service` 做鉴权和参数补写
3. `admin-service` 通过 gRPC 调 `user-service`
4. `user-service` 完成领域逻辑和数据访问
5. 结果返回给后台页面

### 前台请求链路

典型情况是：

1. 前台页面发请求到 `app-service`
2. `app-service` 从 token 中解析当前用户
3. `app-service` 调 `user-service`
4. `user-service` 执行领域逻辑
5. 结果返回给前台页面

### 订单链路

当前仓库里订单链路是一个额外扩展示例：

1. 前台请求先到 `app-service`
2. `app-service` 会先校验用户
3. 然后调 `trade-service`
4. `trade-service` 再调 `book-service`
5. `book-service` 提供库存和书籍数据

这条链路说明：

- 这个仓库已经支持新增独立领域服务
- 但主业务链路仍然以 `user-service(core)` 为中心

## 为什么很多逻辑不该直接放在 BFF

新手最容易犯的错误是：

- 想加后台接口，就直接把逻辑写死在 `admin-service`
- 想加前台接口，就直接把落库逻辑写在 `app-service`

这会带来两个问题：

### 1. 逻辑重复

如果后台和前台都需要这份能力，你最终会复制两套实现。

### 2. 核心规则被终端耦合

本来是领域规则的问题，结果被后台字段、前台字段、某个页面流程绑住了。

## 什么时候放到 `user-service(core)`

满足下面任意一条，优先考虑放到 `user-service`：

- 需要持久化
- 会被多个入口复用
- 涉及实体规则
- 涉及事务
- 不应该依赖某个具体前端页面

常见例子：

- 用户创建规则
- 角色绑定
- 权限校验规则
- 内容实体的增删改查
- 站点和导航的核心结构

## 什么时候放到 `admin-service`

满足下面情况时优先考虑放到 `admin-service`：

- 后台专属接口
- 只有后台才会看到的聚合结果
- 需要 operator、tenant、createdBy 等后台上下文字段补写
- 后台菜单、后台门户、后台审计编排

## 什么时候放到 `app-service`

满足下面情况时优先考虑放到 `app-service`：

- 前台专属接口
- 当前登录用户视角的数据组装
- 前台协议适配
- 靠近前台交互的上传、下载、资料更新接口

## `trade-service` 和 `book-service` 该怎么理解

这两个服务说明仓库已经具备拆独立服务的能力，但你不应该默认每个新领域都照它们去拆。

当前更合理的理解是：

- 它们是独立领域服务示例
- 它们证明“新服务”这条路线是成立的
- 但绝大多数第一阶段业务，仍然更适合先进入 `user-service(core)`

## 当前架构里最需要牢记的边界

### BFF 边界

`admin-service` / `app-service` 主要负责：

- 协议
- 场景
- 上下文
- 编排

### 核心域边界

`user-service` 主要负责：

- 数据拥有权
- 领域规则
- 数据访问
- 事务

### 独立域边界

`trade-service` / `book-service` 主要负责：

- 自己的数据和规则
- 对其他服务暴露清晰的 gRPC 能力

## 开发时最实用的判断口诀

看到一个需求，先问自己四个问题：

1. 这是核心业务规则还是接口适配
2. 这份能力会不会被多个入口复用
3. 这份逻辑需不需要落库或事务
4. 它是不是只属于后台或前台某一端

如果答案偏向“规则 / 复用 / 落库 / 事务”，就先想 `user-service(core)`。

如果答案偏向“接口形状 / 当前用户 / operator / 场景编排”，就先想 BFF。

## 这一步完成后你应该掌握什么

读完本篇，你应该已经能判断：

- 当前哪些服务是主链路
- 哪层该放领域逻辑，哪层该放接口编排
- 为什么 `user-service` 才是当前核心后端主目录
